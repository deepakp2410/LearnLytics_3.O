# Architecture Design Specification (ADS)  
**Project:** LearnLytics 2.0 — Agentic AI Student Learning Behavior Analyzer  
**Role:** Google Staff AI Engineer | Principal AI Architect | Enterprise Agentic Systems Designer  

This document defines the **complete technical blueprint** for LearnLytics 2.0. It is structured for **enterprise-grade Agentic AI engineering**, following Google’s Agentic Engineering principles (Factory Model, Harness, Skills, MCP, A2A, Memory, Evaluation, Observability, Security, Production Readiness).  

---

## SECTION 1 — Harness Specification
### Purpose
The **Harness** is the execution environment and central nervous system of LearnLytics 2.0. It ensures **coordination, observability, and extensibility** across agents, skills, tools, and memory.

### Responsibilities
- Manage execution runtime  
- Route tasks to agents  
- Load and cache skills  
- Manage multi-layer memory  
- Standardize MCP tool access  
- Handle prompts and context  
- Provide observability and evaluation hooks  
- Enforce security and configuration  

### Internal Modules
| Module | Responsibility |
|--------|----------------|
| **Execution Runtime** | Runs workflows end-to-end |
| **Task Router** | Delegates tasks to agents |
| **Prompt Manager** | Manages structured prompts |
| **Skill Loader** | Loads skills from repository |
| **Memory Manager** | Handles multi-layer memory |
| **MCP Tool Manager** | Standardized tool invocation |
| **Agent Runtime** | Executes agent logic |
| **Event Bus** | Event-driven communication |
| **Evaluation Engine** | Monitors performance |
| **Observability Layer** | Logs, metrics, tracing |
| **Configuration Manager** | Centralized configs |
| **Security Layer** | RBAC, encryption, audit |
| **Output Manager** | Final dashboards, reports |

### Lifecycle
1. Initialize Harness modules  
2. Register agents and skills  
3. Accept input (dataset, query)  
4. Route tasks via Task Router  
5. Execute workflows with event-driven orchestration  
6. Log, evaluate, and secure outputs  

---

## SECTION 2 — Orchestrator Specification
### Purpose
The **Orchestrator** is the decision-making brain inside the Harness.

### Responsibilities
- Decompose tasks into subtasks  
- Delegate to appropriate agents  
- Plan workflows dynamically  
- Handle errors and retries  
- Terminate workflows gracefully  

### Decision Process
- Input → Task decomposition → Agent selection → Delegation → Execution → Evaluation → Output  

### Interactions
- **Harness** → Execution runtime, event bus  
- **Memory** → Reads/writes context  
- **Skills** → Loads procedural knowledge  
- **MCP** → Invokes tools  
- **Agent Registry** → Discovers agents  
- **Evaluation** → Monitors quality  

---

## SECTION 3 — Virtual Workforce Design
Each agent is a **specialist**.  

| Agent | Purpose | Inputs | Outputs | Skills | Tools | Memory | Evaluation |
|-------|---------|--------|---------|--------|-------|--------|------------|
| **Data Ingestion Agent** | Load datasets | CSV/Excel | Raw dataset | Dataset Validation | CSV, Excel | Working Memory | Data completeness |
| **Data Cleaning Agent** | Preprocess data | Raw dataset | Clean dataset | Feature Validation | LLM Tool | Session Memory | Error rate |
| **Analytics Agent** | Generate insights | Clean dataset | Stats, trends | Attendance Analysis | Database Tool | Knowledge Memory | Accuracy |
| **ML Prediction Agent** | Forecast outcomes | Features | Predictions | Risk Analysis | ML Tool | Evaluation Memory | Precision/Recall |
| **Recommendation Agent** | Suggest interventions | Predictions | Recommendations | Recommendation Generation | LLM Tool | Knowledge Memory | Human feedback |
| **Visualization Agent** | Build dashboards | Insights | Interactive UI | Dashboard Generation | Visualization Tool | Session Memory | Usability |
| **Report Agent** | Generate reports | Dashboards | PDF | Report Generation | PDF Tool | Knowledge Memory | Completeness |
| **Evaluation Agent** | Monitor system | Logs | Metrics | Evaluation | Search Tool | Evaluation Memory | Workflow success |
| **Memory Agent** | Manage memory layers | Context | Updated state | Procedural Memory | Filesystem Tool | All layers | Consistency |

## SECTION 4 — Agent Cards
Each agent has a **Card** defining identity, capabilities, schemas, and lifecycle.  

Example: **Data Cleaning Agent Card**  
- Identity: `Agent.DataCleaning`  
- Description: Cleans and validates datasets  
- Capabilities: Preprocessing, validation  
- Input Schema: Raw dataset (CSV/Excel)  
- Output Schema: Clean dataset (structured JSON)  
- Security: Restricted to non-PII fields  
- Tools: LLM Tool  
- Memory: Session Memory  
- Events: `CleaningStarted`, `CleaningCompleted`  
- Version: v1.0  
- Owner: Data Engineering Team  
- Lifecycle: Registered → Active → Retired

## SECTION 5 — Skills Architecture
### Skill Ecosystem
Skills = **procedural memory units**.  

Example Skill: **Attendance Analysis**  
- Purpose: Analyze attendance patterns  
- Trigger: Dataset ingestion complete  
- Inputs: Attendance records  
- Outputs: Attendance insights  
- Tools: Database Tool  
- Memory: Knowledge Memory  
- Validation: Cross-check missing values  
- Evaluation: Accuracy of insights  
- Lifecycle: Lazy-loaded, cached, versioned  

**Skill Repository Structure**  
```
/skills
  /attendance_analysis
    SKILL.md
    /scripts
    /references
    /assets
```

## SECTION 6 — Memory Architecture
| Memory Layer | Purpose | Stored Info | Expiration | Persistence |
|--------------|---------|-------------|------------|-------------|
| **Working Memory** | Temporary context | Current task state | Short-lived | Volatile |
| **Conversation Memory** | Dialogue history | Teacher queries | Session | Persistent |
| **Session Memory** | Workflow state | Dataset refs | Session | Persistent |
| **Knowledge Memory** | Institutional knowledge | Student history | Long-term | Persistent |
| **Procedural Memory** | Skills repository | Skills metadata | Versioned | Persistent |
| **Evaluation Memory** | Performance metrics | Accuracy, latency | Rolling window | Persistent |
| **Long-term Memory** | Historical archives | Past workflows | Configurable | Persistent |

## SECTION 7 — MCP Strategy
**MCP Tools** standardize integration.  

| Tool | Purpose | Agent | Permissions |
|------|---------|-------|-------------|
| CSV Tool | Read CSV | Ingestion Agent | Read-only |
| Excel Tool | Read Excel | Ingestion Agent | Read-only |
| Filesystem Tool | File ops | Memory Agent | Read/Write |
| Database Tool | Query DB | Analytics Agent | Read-only |
| Visualization Tool | Dashboards | Visualization Agent | Read-only |
| LLM Tool | Reasoning | Cleaning/Recommendation | Read/Write |
| PDF Tool | Reports | Report Agent | Write-only |
| Search Tool | Evaluation | Evaluation Agent | Read-only |

## SECTION 8 — Agent-to-Agent Architecture
- **Agent Registry** → metadata store  
- **Agent Discovery** → query registry  
- **Delegation** → Orchestrator assigns tasks  
- **Negotiation** → Agents request support  
- **Workflow Transfer** → Ownership shifts via events  
- **State Management** → Session Memory tracks progress  

---

## SECTION 9 — Event-Driven Architecture
Example Event: **PredictionCompleted**  
- Publisher: ML Prediction Agent  
- Subscriber: Recommendation Agent  
- Payload: `{student_id, prediction, confidence}`  
- Purpose: Trigger recommendations  
- Failure: Retry with fallback model  
- Retry: 3 attempts

## SECTION 11 — Observability
- **Execution Timeline** → workflow trace  
- **Distributed Tracing** → agent-level  
- **Skill Timeline** → skill usage logs  
- **Memory Timeline** → updates per layer  
- **Tool Timeline** → MCP calls  
- **Dashboards** → system health  

---

## SECTION 12 — Failure Architecture
- **Invalid Dataset** → fallback validation skill  
- **Tool Failure** → retry with backup tool  
- **Agent Failure** → reassign task  
- **Hallucinations** → human approval gate  
- **Timeouts** → retry policy  

---

## SECTION 13 — Security Architecture
- Authentication → OAuth2  
- Authorization → RBAC  
- Secrets → Vault  
- Data Privacy → GDPR/FERPA compliance  
- Encryption → TLS + AES  
- Audit Logs → immutable storage  
- Secure Prompting → sanitized inputs  

---

## SECTION 14 — Complete Execution Lifecycle
1. Teacher uploads dataset → Event: `DatasetUploaded`  
2. Harness routes to Ingestion Agent → CSV Tool  
3. Cleaning Agent preprocesses → Validation Skill  
4. Analytics Agent generates insights → Attendance Analysis Skill  
5. ML Prediction Agent forecasts outcomes → Risk Analysis Skill  
6. Recommendation Agent suggests interventions → Human-in-the-loop validation  
7. Visualization Agent builds dashboards → Visualization Tool  
8. Report Agent generates PDF → Executive Summary Skill  
9. Evaluation Agent monitors workflow → Evaluation Memory  
10. Harness logs execution → Observability Layer  
11. Teacher receives dashboard + report + evaluation metrics  

---

## SECTION 15 — Build vs Buy Analysis
- **Harness** → Build (custom orchestration)  
- **Skills** → Build (domain-specific)  
- **Visualization** → Buy/Reuse (Power BI, Plotly)  
- **Authentication** → Buy (OAuth2)  
- **Report Generation** → Reuse (PDF libraries)
